# PlantSync 搜索模块需求文档 (v1.0)

## 1. 业务目标
基于本地植物数据库 `rhs_data.plant_rhs_info`，为用户提供高效、多维度的植物搜索与筛选功能，帮助用户快速定位目标植物，并提供对应的养护建议。参考主流电商/内容App的搜索体验，打造结构化清晰、信息丰富的搜索结果页及详情页。

## 2. 页面结构与交互设计 (UI/UX)

### 2.1 搜索触发与交互流程 (Search Trigger & Flow)
*   **触发方式:**
    *   **输入即搜索 (Debounced Input):** 用户在输入框键入文字后，防抖延迟 500ms 自动触发搜索请求，无需手动点击“搜索”按钮。
    *   **历史/热门点击:** 点击历史记录或热门推荐的 Tag 标签，直接将词汇填入搜索框并立即触发搜索。
    *   **回车键触发:** 用户在键盘敲击回车 (Enter) 时立即触发搜索，并收起系统软键盘。
*   **清空与重置:**
    *   输入框右侧提供“清除(x)”按钮，点击清空输入框并退回至**搜索准备页**。
    *   更改快捷筛选条件时，基于当前输入框的关键词重新触发搜索。

### 2.2 搜索准备页 (Search Home)
*   **搜索输入框:**
    *   **输入联想 (Auto-complete):** 用户输入关键词时，实时下拉展示匹配的植物名称或分类，降低输入成本。
*   **历史搜索记录:**
    *   以 Tag 标签形式横向排列，支持一键清空，方便用户回溯搜索路径。
*   **热门搜索推荐:**
    *   基于 `nurseries_count`（售卖苗圃数）或预设配置，展示当前热门植物词汇。

### 2.3 搜索结果页 (Search Results)
**整体布局参考主流 App 结构，强调图片的视觉冲击力和标签的实用性。**

*   **顶部操作栏:**
    *   **快捷筛选 (Quick Filters):** 位于搜索框下方的横向滚动胶囊按钮，放置最常用的 2-3 个维度（如：植物类型、光照、耐寒等级），支持快速单选/多选。
    *   **排序切换 (Sort):** 提供下拉或 Tab 切换（默认相关度，可选热门度、价格）。
    *   **高级筛选入口:** 右侧提供“筛选/更多”按钮，点击从侧边或底部弹出完整抽屉。
*   **结果列表流 (List/Grid View):**
    *   支持单列宽卡片或双列瀑布流（优先推荐双列，突出植物图片 `image`）。
    *   **卡片结构与字段映射 (Card Data Mapping):**
        *   **主图 (Thumbnail):** `image` 字段的缩略图。
        *   **收藏按钮 (Favorite Icon):** 右上角悬浮的 ❤️ 收藏按钮。
        *   **主标题 (Title):** 优先展示 `common_name` (俗名)；若无，则展示 `botanical_name` (学名)。要求最多显示 1 行，超长省略。
        *   **副标题 (Subtitle):** 优先展示 `botanical_name` (学名) 或 `family` (科)。要求字号较小，斜体或灰色展示，最多显示 1 行。
        *   **核心标签区 (Tags):** 动态渲染，带有对应的 Icon。
            *   ☀️ 全日照 / ⛅ 半阴 / ☁️ 全阴 (映射自 `sunlight`)
            *   ❄️ 耐寒等级 (例如: ❄️ H4，映射自 `hardiness_level` 提取的前缀)
            *   🛡️ 低维护 (当 `low_maintenance` 为 1 时展示)
            *   🌸 春季观赏 (映射自 `season_of_interest`)

### 2.4 植物详情页 (Plant Detail Page)
*(深度参考 PictureThis / Ehretia monopyrena 详情设计)*

*   **头部 (Hero Header):**
    *   高清植物大图（支持画廊滑动）。
    *   **主标题:** 俗名 (`common_name`)，大字号加粗。
    *   **别名/其他称呼 (Also known as):** `common_names`。
    *   **学名 (Scientific Name):** `botanical_name`，带斜体，次级字号。
    *   **简介 (Intro):** 提取自 `entity_description` 或 `cultivation` 的一小段描述文本。
*   **基础档案分类卡 (Scientific Classification & Facts):**
    *   以横向滚动或网格卡片形式展示，带有对应的小 Icon：
        *   🌿 **植物类型 (Plant Type):** `plant_type` (如: Shrub, Perennial)
        *   🍃 **叶片类型 (Leaf Type):** `foliage` (如: Evergreen)
        *   📏 **株高 (Plant Height):** `height`
        *   ↔️ **展幅 (Spread):** `spread`
        *   🌸 **开花季节 (Bloom Time):** `season_of_interest`
*   **养护指南 (Care Guide) [核心模块]:**
    *   以清晰的图文/进度条形式展示具体养护动作：
        *   💧 **水分 (Water):** 湿度需求 (`moisture`)。配文：“来源于适度湿润的环境，建议每 1-2 周浇水一次...”。
        *   ☀️ **光照 (Sunlight):** 光照偏好 (`sunlight`)。
        *   🌡️ **温度 (Temperature):** 结合耐寒等级换算 (例如: `hardiness_level` 换算为推荐温度区间)。
        *   🪴 **土壤 (Soil Type):** 土壤类型 (`soil_type`) 及酸碱度 (`ph`)。
        *   ✂️ **修剪 (Pruning):** 修剪说明 (`pruning`)。
        *   🌱 **繁殖 (Propagation):** 繁殖方式 (`propagation`)。
*   **扩展科普与文化 (Culture & Characteristics):**
    *   **产地与分布 (Distribution):** `range`、原生物种标记 (`native`)。
    *   **园林用途 (Garden Use):** `suggested_plant_uses`。
    *   **抗性与安全性 (Resistance & Safety):** 毒性说明 (`toxicity`)、病虫害抗性 (`disease_resistance`, `pest_resistance`)。

## 3. 核心搜索功能与技术实现建议

### 3.1 搜索引擎技术选型 (基于 MySQL)
根据当前决定，前期**直接使用 MySQL 原生查询 (LIKE / Full-Text Search)** 作为搜索引擎底座。

*   **实现方式：**
    *   在 `botanical_name`, `common_name`, `common_names` 等核心检索字段上建立 `FULLTEXT` 索引（如支持）或使用 `LIKE '%keyword%'` 进行模糊匹配。
    *   针对筛选字段（如 `plant_type`, `sunlight`, `hardiness_level` 等）建立联合索引，提升 `WHERE` 条件过滤的查询效率。
*   **优势：** 架构最简单，不引入额外的中间件，开发速度极快，维护成本低。

### 3.2 网站 (Website) 早期落地规划
*   **前端:** 在现有的 `website` 目录下，新建 `search.html` 和 `plant.html`。采用 **原生 HTML/JS + Tailwind CSS (通过 CDN 引入)**，通过 Vanilla JS `fetch` 调用后端接口并动态渲染 DOM。
*   **后端 API:** 在 Node.js (Express) 中新增一套搜索专用的 RESTful API (如 `GET /api/v1/search` 和 `GET /api/v1/plants/:id`)。

### 3.3 客户端 (App) 迁移风险与影响评估
当后续此套技术栈（后端 MySQL 搜索 API）接入到 iOS/SwiftUI 客户端时，将会面临以下体验和技术挑战，需要在后期视情况重构：

1.  **容错性极差 (无拼写纠错 Typo-tolerance):** 
    *   **现象:** App 用户在手机上打字容易出错（如输入 "mosntera" 而非 "monstera"）。MySQL `LIKE` 无法识别拼写错误，将直接返回 0 个结果，极大影响 App 的搜索体验。
2.  **联想词与相关度排序 (Relevance Ranking) 较弱:**
    *   **现象:** App 搜索通常期望最符合意图的词排在最前。MySQL 的全文索引打分机制较为死板，难以实现类似淘宝/小红书那种“按热度+相关度”的复杂动态权重排序。
3.  **不支持拼音/首字母搜索 (针对中文环境):**
    *   **现象:** 如果用户搜索中文名，习惯输入拼音首字母（如 "yy" 搜 "月季"）。MySQL 默认不支持此功能，除非额外增加“拼音列”在数据入库时进行转换存储。
4.  **多条件筛选 (Faceted Search) 性能瓶颈:**
    *   **现象:** 当 App 端的高级筛选面板传入 5-6 个组合条件（如：灌木 + 全日照 + 抗旱 + H4耐寒）时，MySQL 需要执行复杂的联合查询，随着数据量增长，API 响应时间可能超过 200ms，导致 App 界面出现明显的“转圈”卡顿。
    *   **应对:** 后期 App 流量增长或体验要求提高时，建议后端将 MySQL 搜索层替换为 Meilisearch 或 Elasticsearch，iOS 客户端代码本身无需大改（仅更换 API 数据源即可）。

## 4. 异常与边界处理
*   **空结果页 (Zero Results):** 友好提示，并展示“猜你喜欢”或“热门推荐”（提取 `nurseries_count` 最高的 Top N 植物）。
*   **图片缺失处理:** `image` 为空时，使用带有 PlantSync 品牌 Logo 的占位图。
